# WeWeb Cuelume

A [WeWeb](https://www.weweb.io/) custom element that adds curated interaction sounds to your app — synthesized live via the Web Audio API, with no audio files and no runtime dependencies.

## About

This is a thin WeWeb wrapper around [**cuelume**](https://github.com/Danilaa1/cuelume) by [Daniel Belyi](https://github.com/Danilaa1). All the audio synthesis, the fourteen sound recipes, and the `data-cuelume-*` attribute binding are his work — this element only initializes the library inside a WeWeb page and surfaces its controls in the editor.

Sounds are synthesized on the fly rather than loaded, so the whole thing costs no network requests and ships no audio assets. Drop the element once on a page, tag any elements you want to make audible, and you're done.

## What it does

Cuelume is document-wide, not per-element. The library's `bind()` attaches four delegated, capture-phase listeners to the document and resolves attributes at event time. So this element is **an invisible initializer you drop once**, not a button or a sound emitter — you add `data-cuelume-*` attributes to whatever elements you want yourself.

Because the listeners are delegated, elements added later (collection lists, modals, anything conditional) work with no rescan or re-init.

The element binds to the app document via `wwLib.getFrontDocument()` rather than the global `document`, which in the editor is the editor's own document. Binding is deferred until sounds are actually enabled, so an app with sounds off pays no per-event cost.

### Constraint: one element per app

The library's enabled flag is module-level global state, while WeWeb elements are per-instance. Two Cuelume elements — or an SPA navigation where the old one unmounts after the new one mounts — is last-writer-wins, and a muted page can leave audio off after navigating away. Place a single instance on a layout or shared section rather than one per page.

## Usage

Drop the Cuelume element on the page, then add attributes to your other elements via their HTML attributes settings:

| Attribute | Fires on | Default sound |
|---|---|---|
| `data-cuelume-hover` | pointerenter (fine mouse only, throttled to 150ms) | `chime` |
| `data-cuelume-press` | pointerdown | `press` |
| `data-cuelume-release` | pointerup | `release` |
| `data-cuelume-toggle` | click | `toggle` |

```html
<button data-cuelume-press data-cuelume-release>Save</button>
<a data-cuelume-hover="tick">Docs</a>
<button data-cuelume-toggle>Dark mode</button>
```

Give an attribute a value to override the sound; leave it empty for the default. Unrecognized names are a silent no-op.

Available sounds: `chime`, `sparkle`, `droplet`, `bloom`, `whisper`, `tick`, `press`, `release`, `toggle`, `success`, `error`, `page`, `loading`, `ready`. The editor dropdown derives this list from the library's own `sounds` export, so it stays in sync across version bumps.

Browsers won't play anything until the user has interacted with the page at least once (the engine checks `navigator.userActivation`). A hover sound before any click is a deliberate no-op, not a bug.

### Properties

- **Sounds enabled** (bindable) — global on/off. Bind it to a user-preference variable; this is the intended way to mute from a workflow.
- **Play in editor** (default off) — sounds only fire while editing if enabled. No effect on the published app.

### Actions

- **Play sound** — plays any sound from a dropdown. For what attributes can't reach: `success` after an API call resolves, `error` on a failed submit.

## Development

Install dependencies:

```
npm i
```

Serve locally, then add the custom element via the WeWeb editor's developer popup:

```
npm run serve --port=[PORT]
```

Check for build errors before release:

```
npx weweb build --name=cuelume --type=wwobject
```

Note the CLI requires both `--name` and `--type`; `npm run build --name=cuelume` fails because npm doesn't forward the flag to the script. This produces `dist/manager.js`, the editor bundle — `wwEditor` blocks are intentionally retained there. The production bundle that strips them is generated by WeWeb's pipeline on publish, so this build doesn't verify the editor-state-free path.
